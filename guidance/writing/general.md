# 通用写作风格指南

> 此为默认写作风格指南，适用于技术自媒体文章的创作。用户可直接修改本文件，skills 在写作时会自动加载。平台专属规范请在 `platform/` 目录下单独配置，不要写入本文件。

本文档是供 LLM 参考的写作操作手册，指导技术类自媒体文章的创作。目标读者是对 AI、软件工程感兴趣的技术从业者。

---

## 一、整体风格定位

这是一个技术博主的个人分享，不是教材，也不是产品文档。

写文章的时候，脑子里要有一个具体的读者：一个坐在你对面的同行，他有一定基础，但对这个话题还没有系统认知。你的任务不是向他"输出知识点"，而是和他一起把这件事想明白。

语气应该是自然的、口语化的，但不是随意的。可以用"其实"、"说白了"、"你可能会想"这类表达，但不能掉进网络段子和表情包里。

文章不必全程保持客观讲解的口吻。在适当的地方可以用第一人称（"我当时的想法是……"、"我们不妨……"），尤其是经历类、观察类的内容，带一点个人视角会让文章更有温度。

---

## 二、文章类型与对应的导入方式

不同类型的文章，开头的方式也应该不同。不要把"类比开头"当成万能模板，而要根据文章的内容属性来选择最自然的导入方式。

### 概念解释类（如 Prompt Caching）

读者的第一个问题往往是"这是什么，和我有什么关系"。适合先建立背景（这个问题在哪些场景下很重要），再用一个直觉类比打通理解，然后进入技术原理。

### 工程实践/案例分享类（如 Seeing like an agent）

文章本身有一条演进线索：遇到了什么问题 → 第一次怎么解决 → 遇到了新问题 → 又怎么演进。这条线索本身就是最好的导入，不需要额外的类比。直接从"一个设计难题"或"一次失败的尝试"切入，读者的好奇心会自然跟着你往下走。

### 个人经历/项目分享类

从"我遇到了这个问题"或"我当时的想法是……"出发，用第一人称叙事引入。这类文章语气可以更随意，体验和想法的分享本身就是价值所在。

---

## 三、行文结构：用问题驱动，而不是用目录驱动

文章最常见的毛病是把知识按"逻辑层级"排列，结果读起来像目录展开，而不是一次思考过程。

**更好的方式是用"问题链"来驱动行文：** 每一段的结尾，自然引出下一段要回答的问题；每一个新概念的出现，都是因为前面的逻辑走到了这里，读者自然会想问这个问题。

读者的疑问顺序比知识的逻辑顺序更重要。先想清楚读者会依次想知道什么，再按照这个顺序写，而不是按照知识体系的层级来写。

### 承接与过渡

每当从一个段落跳到下一个时，必须有一句承接语，不能硬切。常见的过渡方式：

- 先把上一段提炼成一个推论，再用这个推论引出下面的问题：「理解了这一点，我们自然就能推导出……」
- 用一个反问或悬念：「那它底层是怎么实现的呢？」、「但这里有一个细节值得关注。」
- 用一个转折制造张力：「这套方案早期是奏效的。但随着模型能力持续提升，问题开始反转……」

## 四、信息密度：不要只给结论，要展开推导

技术文章常见的问题是只给结论，不给推导过程。读者看完可能记住了结论，但不知道为什么，也不知道怎么用。

对于重要的技巧或规则，要花几句话讲清楚"为什么是这样"、"如果不这样会怎样"。

**反面示例：**
> 技巧：会话中途不要切换模型。Prompt Caching 是与具体模型绑定的。

**正面示例：**
> Prompt Caching 是和具体模型绑定的，不同模型之间的缓存不能共用。如果你在用 Opus 的对话里已经积累了 10 万 Token 的缓存，这时候切换到 Haiku，之前的缓存全部作废，还得重新为 Haiku 构建一遍——这反而可能比继续用 Opus 还贵。

## 五、语言表达要求

### 不要过度精简

精简是好事，但不能把表达精简到失去人味。以下这类句子要尽量保留：

- 解释性的转折句：「模型有了知识，但得靠 Transformer 来告诉它怎么用」
- 共情性的预判：「你可能会想……但这样做会……」
- 口语化的强调：「这一条很多人容易踩坑」

### 加粗只用于关键术语或刻意强调

加粗不是装饰，不要每段都有加粗词，不要给自然阅读不需要停顿的词加粗。判断标准是：去掉加粗，读者是否还能正常理解——能的话，不加粗。

### 瘦身中英文混排，清除外包翻译感

在技术文章中，适当的英文术语是必要的，但毫无节制的中英文混排会让文章产生严重的"外包机翻感"或"AI 味"。
- **去除不必要的英文括注**：如果中文语境已经能完美传达意思，就不要画蛇添足加上英文（如：不要写"提示词（Prompt）"、"范围蔓延（Scope Creep）"，直接写中文即可）。
- **专业术语自然本土化**：将部分直译感过重的词汇在上下文中替换为地道、自然的中文表达。

### 消除生硬的原生指令，转化为自然语境

使用 LLM 协助写作时，很容易直接堆叠工具或框架的原生指令（如 `create-story`、`code-review` 等干巴巴的符号）。这会打断读者的阅读心流。
- **将指令符号转化为易读的自然语言动作**。例如，不要写"执行 `code-review` 工作流"，而是写"进行毫无妥协的交叉代码审查"。

### 禁止使用的句式

这些句式会让文章立刻有"AI 味"，必须避免：

| 禁止使用 | 可以替换为 |
|----------|------------|
| 以上就是关于 X 的完整梳理 | 自然地收束到核心观点 |
| 希望这篇文章能帮你建立起更系统的认知 | 直接说你希望读者有什么具体收获 |
| 欢迎关注，我会持续分享…… | 更随意的表达，如"感兴趣的话可以关注一下" |
| 总的来说 / 综上所述 | 不用总结性开头，直接写 |
| 本文将从以下几个方面介绍…… | 省略提纲预告，直接进入内容 |

## 六、结尾的写法

结尾是最容易写烂的地方，因为很难不落俗套。

**核心原则：结尾不是总结，是收束。**

好的结尾不是把文章又过一遍，而是在叙述自然走完之后，给一个轻轻的落点。可以是：

- 点出整篇文章背后的核心思路（一句话，不是一段话）
- 提一个读者可以进一步思考的问题
- 说一句真实的、个人化的话

不同文章类型，结尾方式也不同：

- **概念讲解类**：把核心洞察用一句话浓缩，再补一句个人的期许，随意结束。
- **案例/实践类**：用一个带有反思性的观察或者留白收尾，而不是硬性总结。

注意：结尾不能比正文还长，点到即止。

## 七、Markdown 使用规范

- **标题层级**：最多使用到 `###`，通常 `##` 足够。中英文词语之间加空格。
- **加粗**：只加粗关键术语或需要读者刻意停顿的词，不要整句加粗。
- **列表**：步骤、并列要点可以用列表，但不要把本来可以成段的叙述拆成列表。
- **引用块（blockquote）**：只用于官方说明、警告提示或需要视觉区隔的核心观点，不要滥用。
- **分割线（`---`）**：只在文章结尾前用一次，起视觉收束作用，不要频繁分割正文段落。
- **代码格式（backtick）**：只用于代码变量名、文件名、命令，不用于普通词语的强调。

---

## 八、写作流程建议

拿到选题后，推荐按以下顺序写：

1. **判断文章类型**：是概念解释、工程案例，还是个人经历分享？类型不同，导入方式和结构逻辑都不一样。
2. **想清楚读者的疑问顺序**：读者第一个想知道什么，第二个想知道什么。文章的顺序跟着这个走，不跟着知识逻辑层级走。
3. **找到导入的切口**：概念类找直觉类比，案例类找"一个当时的困境"，经历类从"我当时的想法"出发。选最自然的那个，不要强行套用。
4. **正文展开时，每写完一个概念就问自己**：我有没有说清楚"为什么"？读者知道了这个之后，自然会想问什么？带着这个问题继续往下写。
5. **结尾最后写**：写完正文再回来收尾，不要一开始就想好结尾的话——自然流淌出来的结尾才不会有模板感。
