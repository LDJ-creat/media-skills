工程界常说 "缓存决定一切" (Cache Rules Everything Around Me)，这条法则同样适用于 AI Agent。像 Claude Code 这样需要长时间运行的 Agent 产品，正是因为有了 Prompt Caching (提示词缓存) 才具有可行性。它允许我们重用之前请求的首回合计算结果，从而显著降低延迟和成本。

什么是提示词缓存？它是如何工作的？在技术上该如何实现？你可以在 @RLanceMartin 关于提示词缓存以及我们新的自动缓存发布的文章中了解更多。

在 Claude Code，我们的整个架构方案都是围绕提示词缓存构建的。较高的提示词缓存命中率能降低成本，并帮助我们为订阅计划设定更慷慨的速率限制。因此，我们对提示词缓存命中率设置了警报，如果命中率过低，我们会在此申报 SEV (严重事件)。

以下是我们从大规模优化提示词缓存中得出的一些（常常出人意料的）经验教训。

#### 为缓存设计你的提示词结构
提示词缓存通过前缀匹配 (prefix matching) 工作 —— API 会缓存从请求开始到每一个 cache_control 航点的所有内容。这意味着你放置内容的顺序至关重要，你会希望尽可能多的请求共享同一个前缀。

最好的做法是：静态内容放前面，动态内容放后面。对于 Claude Code 来说，结构如下：
1. 静态系统提示词和工具 (全局缓存)
2. Claude.MD (在项目内缓存)
3. 会话上下文 (在会话内缓存)
4. 对话消息

通过这种方式，我们可以最大化共享缓存命中的会话数量。但这其实非常脆弱！以下是我们之前破坏这种排序的几个原因示例：在静态系统提示词中放入精确到秒（深度）的时间戳、非确定性地打乱工具定义顺序、更新工具的参数等。

#### 使用消息来进行更新
有时候你在提示词中放入的信息可能会过期，例如当你加入了时间信息，或者用户修改了某个文件。这时，考虑是否可以在下一轮对话中通过消息将这些信息传进去。在 Claude Code 中，我们在下一条用户消息或工具结果中添加一个 `<system-reminder>` 标签，用于传递给模型的更新信息（比如：现在是星期三了），这有助于保留缓存。

#### 会话中途不要切换模型
提示词缓存是与具体模型绑定的。如果你在使用 Opus 的对话中已经消耗了 10 万 Token，然后切换到 Haiku，你需要重新为 Haiku 构建提示词缓存，这比继续停留在 Opus 还要昂贵。如果你确实需要切换模型，请使用子智能体 (subagents) 来完成特定任务。

#### 绝不要在会话中途添加或移除工具
在对话中途更改工具集会使整个对话的缓存失效，因为工具是缓存前缀的一部分。

#### 计划模式 (Plan Mode) —— 围绕缓存进行设计
为了进入计划模式，我们不会替换现有的工具（这样做会破坏缓存），而是始终将所有工具保留在请求中，并把 `EnterPlanMode` 和 `ExitPlanMode` 本身作为工具。当切换模式时，智能体会收到一条解释指令（例如：仅限探索）的系统消息，但工具的定义永远不会改变。

#### 工具搜索 —— 延迟加载而非直接移除
对于几十个 MCP 工具，我们使用了轻量级的存根（只有一个工具名，并带上 `defer_loading: true`）。只有当模型通过 `ToolSearch` 工具选中它们时，完整的工具 Schema 才会被加载。这保持了缓存前缀的稳定。

#### 分支上下文 —— 压缩
当上下文窗口耗尽时，我们会对对话进行总结（也就是压缩）。为了保证缓存安全，我们复用与父级对话完全相同的系统提示词、工具和历史记录，仅仅将压缩提示词作为一条新的用户消息追加在最后。这确保了前面的缓存前缀能够被重用。

#### 经验总结
1. **提示词缓存是前缀匹配。** 前缀中的任何更改都会使它之后的所有内容缓存失效。
2. **使用消息进行更新，** 而不是去更改系统提示词。
3. **不要在对话中途更改工具或模型。**
4. **像监控正常运行时间 (uptime) 一样监控你的缓存命中率。**
5. **分支操作需要共享其父级的前缀。**

Claude Code 从第一天起就是围绕提示词缓存构建的，如果你也要构建一个 Agent 产品，你同样应该如此。
